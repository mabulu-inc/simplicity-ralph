# T-046: Wire post-iteration metadata updates into orchestrator

- **Status**: TODO
- **Milestone**: 11 — Post-Iteration & Housekeeping
- **Depends**: T-023
- **PRD Reference**: §3.2

## Description

The orchestrator (from T-023) detects commits and pushes, but does not perform the other post-iteration updates specified in PRD §3.2 step 10: backfill SHAs, update costs, and regenerate milestones. The `ralph shas`, `ralph cost --update-tasks`, and `ralph milestones` commands exist as standalone CLI utilities but are never called by the loop.

**Requirements:**

- After a successful commit is detected (HEAD changed), call the core functions from `shas.ts`, `cost.ts`, and `milestones.ts` to:
  1. Backfill the commit SHA into the completed task file
  2. Update the cost field in the completed task file from the log
  3. Regenerate `docs/MILESTONES.md`
- Import and call the underlying functions directly (not via subprocess) — the `run()` exports from each command module accept a `cwd` parameter
- These updates should happen **before** the push, so that the pushed commit includes the metadata updates
- Stage and commit the metadata changes as a separate commit (e.g., `Update task metadata`) so the task commit stays clean
- If any post-iteration update fails, log a warning and continue (do not abort the loop)
- All existing tests must continue to pass

## Hints

- The functions to call are: `run()` from `shas.ts`, `run()` from `cost.ts` (with `['--update-tasks']`), and `run()` from `milestones.ts`
- After calling these, stage the changed files and create a metadata commit before pushing
- Use `LoopGitService` for the metadata commit
- The Verify phase thrashing pattern (running `pnpm check` repeatedly) was identified in T-045 analysis — ensure the agent captures sufficient output on the first run

## Produces

- Updated `src/ralph/commands/loop/orchestrator.ts` with post-iteration metadata updates
- Tests for the post-iteration update flow
