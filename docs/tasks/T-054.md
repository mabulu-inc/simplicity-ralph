## Description

When the loop status transitions to STOPPED, the monitor's staleness timers (last output "ago" and activity "ago") continue ticking on each refresh. This is misleading — if the agent is no longer running, the timers should freeze at the value they had when the status became STOPPED, not keep counting up indefinitely.

**Requirements:**

1. **Freeze staleness timers on STOPPED** — When `status` is `STOPPED`, the "Last output (Xm Ys ago)" and "Activity: Tool (Xs ago)" timers must stop advancing. Display the final elapsed value from when the agent was last active, not the ever-growing delta from `Date.now()`.

2. **All existing tests must continue to pass** — Add tests verifying that timer values are frozen when status is STOPPED and continue ticking when status is not STOPPED.

## Touches

- `src/ralph/commands/monitor.ts`

## Hints

- When computing staleness in `formatMonitorOutput`, check the current status. If STOPPED, use the timestamp of the last log entry (or the stop time) as the reference point instead of `Date.now()`.
- Alternatively, capture the "last active" timestamp when the status transitions to STOPPED and use that as the anchor for all ago calculations.
