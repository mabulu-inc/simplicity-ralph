## Description

The monitor's phase timeline currently has two problems: (1) phases sometimes don't appear at all when RUNNING, and (2) there's no timing information ‚Äî the user can't see how long each phase took or how long the current phase has been running.

**Requirements:**

- **Always show phases when RUNNING**: When status is RUNNING, always display the phase timeline (show all phases as `‚óã` if no phase markers have been found yet)
- **Per-phase durations**: Parse timestamps for each `[PHASE] Entering:` marker from the log. For completed phases (where a subsequent phase exists), show the elapsed duration (e.g., `‚óè Boot (45s) ‚Üí ‚óè Red (1m 12s) ‚Üí ‚óè Green ‚Üí ‚óã Verify ‚Üí ‚óã Commit`)
- **Live timer on active phase**: The current (last) phase should show a live-updating timer that recalculates on each refresh (e.g., `‚óè Green (2:30)`)
- **Faster default refresh**: Change watch mode default interval from 5s to 1s for a responsive live timer
- **Read full log for phases**: The current `readLogTail` (8KB) may miss early phase markers. For phase parsing specifically, scan the full log file for `[PHASE]` markers ‚Äî these are infrequent (at most 5 per iteration) so a targeted scan is feasible. Alternatively, increase the tail size or do a two-pass read (tail for last-output, full scan for phases).
- Remove the separate "Current phase:" line ‚Äî fold that information into the phase timeline itself
- All existing tests must continue to pass

## Hints

- Phase timestamps come from the JSONL entries. Each log line may have a `timestamp` field ‚Äî use `extractTimestampFromJsonLine()` which already exists
- Consider a `PhaseTimestamp` type: `{ phase: string; startedAt: Date | null }[]` ‚Äî parse all phase entries, then calculate durations between consecutive entries
- For the live timer, the last phase's duration is `Date.now() - lastPhaseStartedAt`
- Format durations compactly: `45s`, `1m 12s`, `2:30` ‚Äî keep it short to fit on one line
- The 8KB tail may miss Boot/Red markers if the log is large ‚Äî consider reading the first 4KB too, or scanning specifically for `[PHASE]` lines
- The `formatPhaseTimeline` function currently just shows filled/empty dots ‚Äî it needs to accept timing data
