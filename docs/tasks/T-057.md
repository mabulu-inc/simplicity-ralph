## Description

When a Claude session ends because it hit the max turns limit (`error_max_turns`), the orchestrator only logs `Claude exited with code 1` — the same message as any other failure. The operator has no idea the session ran out of turns vs. crashed, and doesn't know what turn limit was in effect or what complexity tier was assigned.

This matters because a max-turns failure is actionable (bump `--max-turns` or split the task) while a crash is a different problem entirely.

## Requirements

1. **Parse the JSONL log after each session** — read the final `result` entry to extract `subtype` (e.g. `error_max_turns`), `num_turns`, and `stop_reason`.

2. **Log the complexity tier and effective limits at iteration start** — the orchestrator already logs `Starting T-XXX: title` but should also log the assigned tier and effective max turns/timeout, e.g. `  Complexity: standard (75 turns, 900s timeout)`.

3. **Log a specific message when max turns is exceeded** — instead of the generic exit-code error, log something like: `[Iteration N] T-XXX exhausted all 75 turns (standard tier) without completing. Consider increasing --max-turns or splitting the task.`

4. **Log turns used on success too** — on a successful iteration, include turns used, e.g. `[Iteration N] Commit detected: abc1234 (used 42/75 turns)`.

5. **All existing orchestrator tests must continue to pass.**

## Produces

- Updated `src/ralph/commands/loop/orchestrator.ts` with improved logging
- New helper to parse the final result entry from a JSONL log file
- Tests for the JSONL result parser and the new log output

## Hints

- The JSONL log's last line with `"type":"result"` contains `"subtype":"error_max_turns"`, `"num_turns"`, and `"stop_reason"`
- The `scaleForComplexity` return value already has `tier`, `maxTurns`, and `timeout` — just log it
- Avoid reading the entire log into memory — read from the end or stream the last few lines
