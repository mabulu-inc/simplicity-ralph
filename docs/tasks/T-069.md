# T-069: Smarter dependency parsing and diagnostic "no eligible task" messages

- **Status**: DONE
- **Milestone**: 8 — Robustness
- **Depends**: none
- **PRD Reference**: §3
- **Complexity**: standard
- **Completed**: 2026-03-18 09:55 (5m duration)

## Description

When `findNextTask` returns nothing, the loop prints a generic message:

> No eligible task found (remaining tasks may be blocked or have unmet dependencies)

This gives no clue _why_ tasks are ineligible. In practice, a common cause is a malformed Depends field (e.g., `(none)` or `—` instead of `none`) which gets parsed as a dependency on a nonexistent task, silently blocking every task in the project.

Two improvements:

### 1. Robust dependency parsing

`parseDeps` currently only recognizes the exact string `none`. It should also treat common variants as "no dependencies":

- `(none)`, `—`, `–`, `-`, empty string, whitespace-only

### 2. Diagnostic output when no task is eligible

When `findNextTask` returns `undefined` and there are remaining TODO tasks, print a breakdown explaining why each TODO task was skipped. For each ineligible TODO task, show:

- Whether it's marked blocked
- Which dependencies are unmet (and whether each unmet dep exists as a task at all vs. is a phantom reference)

Example output:

```
No eligible task found. 3 TODO tasks remain:
  T-087: blocked by unmet deps: (none) [unknown task — possible typo in Depends field]
  T-088: blocked by unmet deps: (none) [unknown task — possible typo in Depends field]
  T-095: blocked by unmet deps: T-087 [TODO], T-088 [TODO]
```

This should only appear in verbose mode (`-v`) to keep default output clean. In non-verbose mode, keep the existing one-liner but append a hint: `(re-run with -v for details)`.
